## 四、Pull 模型原理

### 4.1 Prometheus Pull 模型
Prometheus 采用 **Pull（拉取）模型** 获取监控数据，而非传统的 Push（推送）模型。

**Pull 模型的优势：**

1. **集中控制**: Prometheus 主动控制数据采集频率和目标，便于统一管理
2. **服务发现**: 通过 Kubernetes 服务发现机制，自动发现新的监控目标
3. **健康检查**: 每次拉取数据时，Prometheus 可以判断目标是否存活
4. **减少依赖**: 被监控服务无需实现推送逻辑，只需暴露 /metrics 接口
5. **易于调试**: 可以直接访问目标的 /metrics 接口查看原始数据

**工作流程：**
1. Prometheus 配置 scrape_configs 定义采集目标
2. Prometheus 定期（默认 15s）向目标发送 HTTP GET 请求
3. 目标返回 Prometheus 格式的指标数据
4. Prometheus 解析并存储数据到 TSDB（时间序列数据库）
5. Grafana 通过 Prometheus API 查询数据并可视化

## 五、核心指标含义

### 5.1 node_cpu_seconds_total
```
指标名称: node_cpu_seconds_total
含义: 节点 CPU 累计使用时间（秒）
类型: Counter（计数器）
```

**解释：**
- 该指标记录了 CPU 在各种模式下累计运行的时间
- 常见模式包括：user（用户态）、system（内核态）、idle（空闲）、iowait（IO等待）等
- 使用 `rate()` 函数计算速率，可以得到 CPU 使用率

**示例 PromQL：**
```promql
# 计算 CPU 使用率（排除空闲时间）
avg(rate(node_cpu_seconds_total{mode!="idle"}[5m])) by (instance)
```

### 5.2 container_memory_working_set_bytes
```
指标名称: container_memory_working_set_bytes
含义: 容器工作集内存（字节）
类型: Gauge（仪表盘）
```

**解释：**
- 这是衡量 Pod 内存使用的 **最关键指标**
- 工作集内存 = 当前活跃使用的内存 + 最近使用但可能被回收的内存
- 当工作集内存超过容器内存限制时，容器会被 OOM Killed
- 比 container_memory_usage_bytes 更准确反映实际内存压力

**示例 PromQL：**
```promql
# 查询各容器的内存使用
container_memory_working_set_bytes{container!="",container!="POD"}
```

### 5.3 kube_pod_status_phase
```
指标名称: kube_pod_status_phase
含义: Pod 当前状态阶段
类型: Gauge（仪表盘）
值: 0 或 1（表示是否处于该阶段）
```

**解释：**
- Pod 可能的状态阶段：
  - Pending：等待调度
  - Running：正常运行
  - Succeeded：成功完成
  - Failed：失败终止
  - Unknown：状态未知
- 该指标为每个 Pod 记录其当前所处的阶段（值为 1）

**示例 PromQL：**
```promql
# 查询所有 Running 状态的 Pod
kube_pod_status_phase{phase="Running"}
```

## 六、监控系统的价值

### 6.1 运维价值
1. **实时监控**: 及时发现集群异常，如节点 CPU 过载、Pod 内存泄漏
2. **历史分析**: 通过历史数据分析性能瓶颈和资源使用趋势
3. **告警通知**: 配置告警规则，自动通知运维人员处理异常

### 6.2 资源优化
1. **容量规划**: 根据历史数据预测资源需求，合理扩容
2. **成本控制**: 发现资源浪费，优化 Pod 配置降低成本
3. **性能调优**: 定位性能瓶颈，针对性优化

### 6.3 故障排查
1. **快速定位**: 通过 Dashboard 快速发现异常组件
2. **根因分析**: 结合多个指标分析故障根本原因
3. **验证修复**: 监控修复效果，确保问题解决
